## 1. 작업 배경

Gemini API 기반 AI 여행 코스 생성 기능이 로컬에서 정상 호출되기 시작하면서, 실제 사용자 흐름 기준의 테스트를 진행했습니다.

테스트 중 다음 문제가 확인되었습니다.

```text
1. Gemini가 관광공사 API에 등록되지 않은 장소 위주로 코스를 생성함
2. TourAPI contentId가 없는 AI 생성 장소는 위시리스트 카드로 저장되지 않고 메모에만 저장됨
3. 위시리스트 폴더 기반으로 코스를 생성해도 기존 폴더가 아니라 동일한 새 폴더가 다시 생성됨
4. AI 플래너 입력 화면에서 지역/예산/생성 기준에 대한 설명이 부족해 사용자가 입력 기준을 이해하기 어려움
5. Firebase API key 제한 설정으로 로그인 API가 차단되는 문제가 발생함
```

오늘 작업은 AI 플래너 기능을 공모전 제출 관점에서 더 설득력 있게 만들기 위해, **관광공사 API 데이터 우선 활용**, **위시리스트 폴더 기반 코스 생성**, **기존 폴더 저장 보정**, **입력 안내 강화**를 중심으로 진행했습니다.

## 2. Firebase API key 및 로그인 트러블슈팅

### 발생 증상

로그인 시 다음 오류가 발생했습니다.

```text
Firebase: Error (auth/requests-to-this-api-identitytoolkit-method-google.cloud.identitytoolkit.v1.authenticationservice.signinwithpassword-are-blocked.)
POST https://identitytoolkit.googleapis.com/... 403 Forbidden
```

### 원인 분석

처음에는 Gemini API key 문제와 혼동될 수 있었으나, 실제 원인은 Firebase Authentication 로그인 API가 사용하는 **Identity Toolkit API**가 API key 제한 설정에 의해 차단된 것이었습니다.

확인한 내용:

```text
Google Cloud Credentials에 Browser key (auto created by Firebase)가 2개 존재
2026년에 생성된 Browser key는 로그인 호출에서 403 발생
2019년에 생성된 Browser key를 .env의 VITE_FIREBASE_API_KEY로 사용하자 로그인 정상 동작
Firebase 콘솔에 표시되는 Web App API key와 실제 로컬 .env에서 사용하는 key가 다를 수 있음
Firebase Auth 로그인은 Gemini API key가 아니라 Firebase Browser key와 Identity Toolkit API 제한 설정의 영향을 받음
```

### 판단 기준

```text
Gemini 호출 실패
- generativelanguage.googleapis.com
- VITE_GEMINI_API_KEY
- Gemini API, quota, 모델명, 서버 상태 확인

Firebase 로그인 실패
- identitytoolkit.googleapis.com
- VITE_FIREBASE_API_KEY
- Browser key(auto created by Firebase), Identity Toolkit API 제한 확인
```

### 조치 결과

```text
2019년 Browser key를 VITE_FIREBASE_API_KEY로 적용한 뒤 로그인 정상화
2026년 Browser key는 삭제하지 않고 사용 중지 또는 분리 검토 대상으로 판단
서비스 계정에 바인딩된 Gemini API key는 웹사이트 HTTP referrer 제한을 적용할 수 없음을 확인
```

```text
설정한 조건으로 새 코스 생성하기
```


#### 위시리스트 폴더 기반으로 코스 생성하기

```text
사용자가 기존 위시리스트 폴더를 선택
선택한 폴더에 저장된 여행지 목록과 선택 개수를 표시
폴더 내 장소를 기본적으로 전체 선택
필요 시 장소별 포함/제외 가능
Gemini는 선택한 장소를 주요 방문지로 우선 배치
저장 시 기존 폴더에 AI 메모와 체크리스트만 추가
동일한 이름의 새 폴더를 다시 생성하지 않음
```

## 6-1. 인원 수 및 총예산 프롬프트 반영

### 작업 배경

예산 설명을 `1일 1인 기준`으로 정리한 뒤, 실제 사용자가 2명 이상으로 여행할 경우 Gemini가 전체 예산 감각을 함께 고려하는지 점검했습니다.

확인 결과 기존 AI 플래너는 `동행 유형`만 가지고 있었고, 실제 `인원 수`를 별도로 입력하거나 총예산을 계산하는 로직은 없었습니다.

기존 전달 정보:

```text
동행 유형: 친구
예산 수준: 보통
예산 기준: 1일 1인 3만~8만 원
```

문제점:

```text
2명, 3명, 가족 단위 여행이어도 Gemini에는 실제 인원 수가 전달되지 않음
예산 기준은 1인 기준으로 명확하지만, 전체 여행 비용 감각은 모델이 추론해야 하는 상태
동행 유형과 실제 인원 수가 분리되어 있지 않아 예산 조정 근거가 부족함
```

## 8. AI 생성 장소 표시 방식 논의

### 확인한 문제

관광공사 API에 등록되지 않은 AI 생성 장소는 `contentId`가 없기 때문에 기존 여행지 상세 페이지(`/explore/{contentId}`)로 연결할 수 없습니다.

이 장소를 억지로 위시리스트 카드처럼 저장하면 다음 문제가 생길 수 있습니다.

```text
VIEW_DATA 클릭 시 상세 페이지에 필요한 TourAPI detailCommon/detailIntro/detailImage 데이터를 조회할 수 없음
지도, 주소, 이미지, 입장료 등 신뢰 가능한 데이터가 부족함
TourAPI 장소 카드와 AI 생성 장소 카드의 정보 신뢰도가 섞임
```

### 제안 방향

AI 생성 장소를 기존 위시리스트 카드로 억지 저장하기보다, 별도 `AI 코스 기록` 또는 `AI Course Detail` 형태로 분리하는 방향을 제안했습니다.

권장 구조:

```text
TourAPI 장소
- 위시리스트 카드
- VIEW_DATA
- 관광공사 상세 정보 페이지 연결

AI 생성 코스
- AI 코스 기록 카드
- VIEW_COURSE
- title, summary, days, items, tips, checklist 기반의 게시글형 상세 화면 제공
```

후속 작업 후보:


#### ESLint가 .obsidian 플러그인 파일까지 검사하는 문제

```text
원인:
문서 폴더 내부에 Obsidian 플러그인 번들 JS 파일이 포함되어 있었고,

해결:

- **/.obsidian/**
- **/.claude/**
- **/.claudian/**
```

수정 후 전체 lint는 error 없이 통과했으며, 기존 React Hook warning 11개만 남아 있습니다.

## 21. AI 여행 코스 저장 위치 보정

### 작업 배경

AI 여행 코스를 위시리스트 폴더에 저장했을 때, 코스 전체 설명 본문이 `wishlistNotes`의 `MEMO` 타입으로 저장되어 마이페이지의 `FOLDER_NOTES > MEMO` 탭에 긴 텍스트로 노출되는 문제가 확인되었습니다.

```text
문제:
AI 코스는 사용자가 직접 작성한 폴더 메모가 아닌 생성 결과 데이터임
그런데 기존 저장 로직이 코스 제목, 요약, Day별 일정 전체를 MEMO 노트로 저장함
결과적으로 폴더 메모 탭이 AI 코스 본문으로 채워져 사용자가 작성한 메모와 구분되지 않음
```

### 수정 내용

```text
saveAiTripToFolder 저장 로직 수정
AI 코스 전체 본문을 wishlistNotes/MEMO로 저장하던 처리 제거
AI 코스 원본 데이터는 users/{uid}/aiTripPlans/{planId} 노드에 별도 저장
aiTripPlans에는 folder_id, title, summary, saveGuide, days, created_at 저장
기존 위시리스트 카드 저장과 체크리스트 저장은 유지
TourAPI ID가 없는 추천 장소만 있는 경우에도 메모가 아닌 코스 데이터와 체크리스트로 저장되도록 안내 문구 변경
```

### 저장 구조

```text
users/{uid}/wishlistFolders/{folderId}
  - 폴더 기본 정보

users/{uid}/wishlists/{wishlistId}
  - TourAPI contentId가 있는 추천 장소 카드

users/{uid}/wishlistNotes/{noteId}
  - 사용자가 직접 작성한 메모
  - AI 코스 saveGuide.checklist 기반 체크리스트

users/{uid}/aiTripPlans/{planId}
  - AI 여행 코스 생성 결과 원본
  - folder_id로 연결
```

### 참고

이번 수정은 새로 저장되는 AI 코스부터 적용됩니다. 이미 Firebase에 저장된 `[AI 여행 코스]` 메모 데이터는 기존 데이터이므로, 필요 시 별도 정리 작업으로 삭제하거나 `aiTripPlans`로 이전하는 마이그레이션을 진행해야 합니다.

## 22. AI 여행 코스 문서형 표시 및 메모 분리 보정

### 작업 배경

AI 여행 코스 저장 결과가 폴더의 `FOLDER_NOTES > MEMO` 영역에 긴 본문으로 노출되어, 사용자가 직접 작성하는 메모와 CodeTrip이 생성한 여행 일정 데이터가 섞여 보이는 문제가 있었습니다.

또한 TourAPI contentId가 없는 추천 장소가 포함된 경우 위시리스트 카드로 저장되지 못하고, 코스 전체가 메모처럼 들어가 사용자가 실제 일정 결과를 확인하기 어려웠습니다.

### 수정 내용

```text
저장 구조:
- AI 코스 원본은 users/{uid}/aiTripPlans/{planId}에 별도 저장
- 저장 필드: folder_id, user_id, title, summary, saveGuide, days, created_at
- TourAPI contentId가 있는 장소는 기존 wishlists 카드로 저장
- 여행 준비 항목은 기존 wishlistNotes/CHECKLIST로 저장
- AI 코스 전체 본문은 wishlistNotes/MEMO에 저장하지 않도록 수정
```

```text
화면 표시:
- 마이페이지에서 폴더 선택 시 연결된 AI 코스를 함께 조회
- AI 코스는 AI_COURSE.md 문서형 카드로 표시
- 제목, 요약, Day별 테마, 시간대별 장소, 주소, 추천 사유를 게시글 형태로 출력
- 기존에 잘못 저장된 [AI 여행 코스] 메모는 메모 목록에서 숨김 처리
- 저장된 여행지 카드가 없고 AI 코스/체크리스트만 있는 경우 빈 상태 안내 문구를 한 줄로 정리
```

### 변경 파일

```text
src/pages/AiPlanner.jsx
```

### 트러블슈팅 메모

기존 Firebase에 이미 저장된 `[AI 여행 코스]` 메모 데이터는 삭제하지 않고 화면에서만 숨김 처리했습니다. 사용자 데이터 삭제는 별도 확인이 필요한 작업이므로, 추후 필요하면 `wishlistNotes`에서 해당 prefix를 가진 MEMO 데이터를 `aiTripPlans`로 이전하거나 삭제하는 마이그레이션을 별도 이슈로 진행하는 것이 좋습니다.
