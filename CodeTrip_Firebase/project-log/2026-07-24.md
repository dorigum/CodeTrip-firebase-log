# CodeTrip 개발 로그 - 2026-07-24

## 1. 작업 개요

Gemini API의 기본 모델을 Flash-Lite로 변경하고 일시적인 API 장애에 대응하는 재시도 로직을 추가했습니다.
마이페이지에서는 AI 여행 코스를 일반 메모와 분리해 마크다운 문서 형태로 조회할 수 있도록 개선했으며, 기존 방식으로 저장된 AI 코스 메모를 새 데이터 구조로 전환하는 마이그레이션 기능도 추가했습니다.

## 2. Gemini Flash-Lite 전환

- 기본 모델을 `gemini-3.5-flash`에서 `gemini-3.5-flash-lite`로 변경
- `README.md`의 `VITE_GEMINI_MODEL` 예시 최신화
- 로컬 `.env`의 모델 설정 변경
- API key 값은 변경하지 않음
- Flash-Lite 요청에서 지원하지 않는 `temperature` 옵션 제거
- JSON 응답 형식을 위한 `responseMimeType: application/json` 설정은 유지

## 3. Gemini API 재시도 처리

다음과 같이 일시적으로 복구될 수 있는 응답만 최대 2회 재시도하도록 구성했습니다.

- `408 Request Timeout`
- `429 Too Many Requests`
- `500` 이상 서버 오류

재시도 간격은 1초, 2초로 증가하며 최대 249ms의 임의 지연을 더합니다. 인증, 권한, 잘못된 요청과 관련된 `400`, `401`, `403` 오류는 재시도하지 않고 즉시 사용자 안내로 처리합니다.

## 5. 레거시 AI 메모 마이그레이션

과거 버전에서는 AI 코스 전체 내용이 `wishlistNotes`의 `MEMO` 항목에 `[AI 여행 코스]` 접두어와 함께 저장됐습니다. 이 데이터가 일반 메모 목록을 차지하지 않도록 다음 흐름을 추가했습니다.

1. 선택한 폴더에서 `[AI 여행 코스]`로 시작하는 메모 감지
2. 일반 메모 목록에서는 해당 항목을 분리
3. 사용자에게 기존 AI 코스 발견 안내 표시
4. `Convert_to_course` 버튼과 확인 팝업 제공
5. 사용자 확인 후 `aiTripPlans` 노드로 일괄 전환
6. 새 코스 문서 저장과 기존 메모 제거를 하나의 Firebase 다중 경로 업데이트로 처리
7. 전환 완료 메시지는 초록색 성공 토스트와 체크 아이콘으로 표시

전환된 문서에는 다음 정보가 보존됩니다.

- 기존 메모 원문
- 기존 생성일
- 원본 메모 ID
- 마이그레이션 일시
- 원본 데이터 유형을 나타내는 `source: legacy_wishlist_note`

레거시 문서는 구조화된 `days` 데이터가 없으므로 상세 화면의 `LEGACY_COURSE_CONTENT` 영역에서 원문을 줄바꿈과 함께 표시합니다. 데이터는 사용자의 명시적인 확인 전에는 변경하지 않습니다.

## 10. 위시리스트 폴더 날짜 입력 검증

폴더 생성 및 수정 화면의 날짜 입력에서 브라우저가 5자리 이상의 연도를 허용할 수 있는 문제를 수정했습니다.

- 폴더 생성 및 수정 저장 직전에 날짜 형식을 다시 검사
- 종료일이 시작일보다 빠른 경우 저장 차단

## 12. AI 플래너 안내 색상 및 폴더 조건 자동 반영

AI 플래너에서 `혼자`를 선택했을 때 인원 수가 1명으로 자동 조정되는 안내는 오류가 아닌 정보성 동작입니다. 사용자가 잘못 입력한 것으로 오해하지 않도록 빨간색 오류 토스트 대신 프로젝트 기본 청록색 정보 토스트와 `info` 아이콘을 사용하도록 변경했습니다.

위시리스트 폴더 기반 코스 생성 시 선택한 폴더의 메타데이터와 저장 장소를 하단 여행 조건에 자동 반영하도록 보완했습니다.

- 폴더에 지역 필드가 있으면 해당 값을 `REGION`에 우선 반영
- 별도 지역 필드가 없으면 폴더에 저장된 장소 주소의 시·도를 집계해 대표 지역 추론
- `울산광역시`, `서울특별시` 등의 행정구역 명칭은 `울산`, `서울`처럼 검색에 적합한 값으로 정규화
- 폴더의 시작일과 종료일을 포함 일수로 계산해 `DAYS`에 반영
- 시작일만 설정된 폴더는 1일 일정으로 처리
- AI 플래너의 현재 최대 범위에 맞춰 자동 반영 일수는 최대 5일로 제한

자동 반영 후에도 사용자가 `REGION`과 `DAYS`를 직접 수정할 수 있도록 입력 필드는 기존 방식으로 유지했습니다.

## 15. Firebase Auth 세션 만료 보강

Firebase Auth의 기본 인증 상태는 개발 서버 재실행이나 Firebase Hosting 재배포만으로 자동 로그아웃되지 않습니다. 사용자 인증이 브라우저 저장소에 복구되기 때문입니다.

CodeTrip은 로그인 시점부터 최대 2시간만 유지되는 절대 만료 세션으로 관리하도록 기존 로직을 보강했습니다.

- 로그인 성공 시 `trip_auth_expires_at`에 로그인 시각 기준 2시간 후를 저장
- 앱 최초 실행 시 저장된 사용자 정보를 사용하기 전에 만료 여부를 즉시 검사
- 만료된 `trip_user`, `trip_token`, 만료 시각 정보를 즉시 제거
- Firebase Auth 인증 상태도 함께 `signOut`
- 만료 시각에 맞춘 타이머 외에 1분 주기 보조 검사 추가
- 백그라운드 탭 복귀와 브라우저 창 포커스 시 만료 여부 재검사
- React 개발 모드 재마운트 시 이전 인증 리스너와 세션 감시 이벤트를 안전하게 정리

로그인 후 2시간 이내의 새로고침·로컬 서버 재실행·재배포 접속에서는 세션이 유지됩니다. 2시간이 지난 뒤에는 서버 상태와 관계없이 앱 재진입 또는 실행 중 감시 로직을 통해 자동 로그아웃됩니다.

인증 스토어 ESLint 검사와 프로덕션 빌드 성공 후 Firebase Hosting에 세션 만료 보강 버전을 추가 배포했습니다.

## 16. AI 코스 저장 시 로그인 상태 불일치 수정

### 증상

- 헤더에는 `AUTHENTICATED`와 사용자 프로필이 표시됨
- AI 여행 코스의 `SAVE FOLDER` 실행 시 `로그인이 필요합니다.` 오류 발생

### 원인

앱 시작 시 Firebase Auth의 인증 복구가 완료되기 전에 `localStorage`의 `trip_user`만으로 Zustand의 `isLoggedIn`을 `true`로 초기화하고 있었습니다.

이 상태에서는 화면과 보호 라우트가 회원용 UI를 먼저 표시하지만, 저장 API가 사용하는 `firebaseAuth.currentUser`는 아직 `null`일 수 있습니다. 그 결과 UI 로그인 상태와 실제 Firebase 인증 상태가 어긋났습니다.

### 수정

- 인증 스토어의 초기 `user`와 `isLoggedIn`을 비로그인 상태로 시작
- Firebase `onAuthStateChanged` 확인이 끝난 뒤에만 로그인 상태 확정
- 보호 라우트에서 `isLoading`을 `isLoggedIn`보다 먼저 검사
- 인증 확인 중에는 AI 플래너 등 회원 전용 화면 대신 세션 확인 로딩 UI 표시
- 헤더와 사이드바도 `isLoggedIn && user` 조건을 모두 만족할 때만 사용자 프로필 표시
- 인증 확인 중 헤더·사이드바에는 레이아웃 이동을 줄이는 로딩 표시 적용
- 알림 조회도 실제 인증이 확정된 경우에만 실행

## 17. 첫 로그인 Permission denied 및 AI 코스 저장 인증 오류 수정

### 증상

- 첫 번째 로그인 요청에서 `Permission denied` 표시
- 로그인 버튼을 두 번째 눌러야 정상 로그인
- 로그인 화면이 표시된 뒤에도 AI 여행 코스 저장 시 `로그인이 필요합니다.` 오류 발생
- 위시리스트 동기화와 AI 코스 저장이 동일한 로그인 오류를 출력

### 원인

2시간 세션 만료 로직과 Firebase Auth 로그인 완료 이벤트의 실행 순서가 충돌했습니다.

1. 사용자가 로그인 버튼 클릭
2. Firebase `signInWithEmailAndPassword` 성공
3. Zustand의 `onAuthStateChanged`가 로그인 화면 처리보다 먼저 실행
4. 아직 `trip_auth_expires_at`이 저장되지 않아 신규 로그인을 만료 세션으로 판단
5. 인증 리스너가 Firebase 사용자를 즉시 로그아웃
6. 동시에 실행 중이던 사용자 프로필 조회가 Realtime Database Rules에서 `Permission denied`
7. 실제 `firebaseAuth.currentUser`가 제거되어 위시리스트와 AI 코스 저장도 로그인 오류 발생

### 수정

- 로그인 요청 시작 전에 2시간 만료 시각을 먼저 저장
- 인증 진행 여부를 별도 상태로 관리
- 로그인 처리 중 Firebase Auth가 일시적으로 `null`을 전달해도 자동 로그아웃하지 않음
- 로그인 실패 시 Firebase Auth, 사용자 정보, 토큰, 만료 시각을 모두 롤백
- 로그인 성공 후 인증 진행 상태를 종료하고 기존 세션 감시 시작
- 로그인 요청 중 버튼 비활성화 및 `Authenticating...` 진행 상태 표시
- 중복 로그인 요청 차단
- 인증 권한 오류 발생 시 영문 `Permission denied` 대신 세션 재확인 안내 표시

npm run build

## 18. 위시리스트 폴더 기반 코스의 지역 자동 반영 보완

### 증상

- 위시리스트 폴더 기반 생성 모드에서 울산 여행지가 담긴 폴더를 선택해도 `REGION` 입력값은 기본값인 `서울`로 유지되었습니다.
- 생성된 AI 여행 코스 자체는 간절곶, 간절곶 소망길, 간절곶소망우체통, 강동몽돌해변 등 선택한 폴더의 장소를 정상 반영했습니다.
- 따라서 추천 데이터는 정상이고, 폴더 선택 시 입력 폼에 지역을 표시하는 동기화 로직만 누락된 상태였습니다.

### 원인

- 일부 기존 위시리스트 항목에는 주소(`addr1`)가 저장되어 있지 않았습니다.
- 기존 폴더에도 별도의 `region` 메타데이터가 없는 경우가 있었습니다.
- 기존 지역 추론은 폴더의 지역 필드 또는 장소 주소만 확인하여, `울산 야구보러 갔다가 뭐하지...?`처럼 폴더명에 포함된 지역을 활용하지 못했습니다.

### 수정

- 폴더 선택 시 지역을 다음 우선순위로 추론하도록 보완했습니다.
  1. 폴더에 명시적으로 저장된 지역
  2. 폴더 내 장소 주소에서 가장 많이 확인되는 지역
  3. 폴더명에 포함된 시·도 지역명
- `울산광역시`, `서울특별시`, `경기도`와 같은 행정구역 명칭을 화면 입력값인 `울산`, `서울`, `경기` 형태로 정규화했습니다.
- 새로 위시리스트에 저장하는 장소는 주소 정보도 함께 보존하도록 저장 구조를 보완했습니다.
- 위시리스트 모달에서 장소 주소가 스토어와 API 계층까지 전달되도록 수정했습니다.

### 기대 동작

- 기존 폴더를 다시 선택하면 폴더명이나 저장 장소 주소를 기준으로 `REGION`이 자동 변경됩니다.
- 울산 폴더를 선택한 경우 `REGION`은 `울산`으로 표시됩니다.
- 폴더 장소를 우선 반영하는 기존 코스 생성 동작은 그대로 유지됩니다.

## 19. 저장 장소 주소 기반 세부 지역 자동 설정

### 개선 배경

폴더명은 사용자가 자유롭게 작성하므로 `고양`과 `고양이`처럼 지명과 일반 단어를 잘못 구분할 가능성이 있습니다. 이에 따라 폴더명을 주요 지역 판단 기준으로 사용하지 않고, 폴더에 실제로 저장된 여행지 주소를 우선 사용하도록 지역 추론 방식을 개선했습니다.

### 적용 기준

1. 폴더에 저장된 장소들의 주소 확인
2. 기존 데이터에 주소가 없으면 관광공사 상세 API를 `contentId`로 조회하여 주소 보완
3. 여러 장소 주소에서 공통으로 확인되는 행정구역 추출
4. 주소를 확보할 수 없는 경우에만 폴더 지역 메타데이터 또는 폴더명을 보조 수단으로 사용

예를 들어 동탄 장소들의 주소가 모두 `경기도 화성시`로 시작하면 REGION에는 `경기도 화성시`가 반영됩니다. 실제 주소 데이터에 더 세부적인 행정구역이 공통으로 존재할 때만 해당 범위까지 표시합니다.

### 추가 보완

- 주소 상세 조회는 기존 API 캐시 계층을 사용하여 반복 호출을 줄였습니다.
- 사용자가 폴더 선택을 빠르게 변경해도 이전 비동기 주소 조회 결과가 현재 선택을 덮어쓰지 않도록 요청 순서를 검증합니다.
- 주소 기반 판단이 가능하면 폴더명에 포함된 단어는 지역 추론에 사용하지 않습니다.

## 20. 동탄 생활권 보완 및 홈 AI 계획 생성 진입점 추가

### 동탄 지역 표시 보완

관광공사 상세 주소가 `경기도 화성시`까지만 제공되는 동탄 장소는 주소만으로 생활권을 구분할 수 없었습니다. 주소를 기본 지역으로 사용하되, 다음 조건을 모두 만족하는 경우 저장 장소명에서 확인되는 생활권을 덧붙이도록 보완했습니다.

## 21. AI 여행 코스 문서 관리 기능 고도화

### 작업 배경

기존 AI 여행 코스는 위시리스트 폴더에 문서 형태로 저장하고 상세 내용을 조회할 수 있었지만, 저장 후 제목이나 요약을 수정하거나 불필요한 코스를 삭제할 수 없었습니다. 또한 같은 조건으로 코스를 다시 만들려면 플래너의 입력값을 처음부터 다시 설정해야 했습니다.

이에 따라 저장된 AI 코스를 일반 메모가 아닌 독립적인 여행 코스 문서로 관리할 수 있도록 수정·삭제·재생성 흐름을 추가했습니다.

### AI 코스 문서 수정

- `AI_COURSE_DETAIL.md` 상세 화면에 `Edit_document` 기능 추가
- 코스 제목과 요약을 상세 화면에서 직접 수정
- 제목을 비워둔 상태로 저장하지 못하도록 입력값 검증
- 저장 중에는 중복 요청을 방지하기 위해 버튼 비활성화
- 수정 성공 시 상세 화면과 폴더의 코스 목록을 즉시 갱신
- Firebase `aiTripPlans/{planId}`에는 제목, 요약과 `updated_at`만 부분 업데이트

### AI 코스 문서 삭제

- 상세 화면에 `Delete_document` 기능 추가
- 프로젝트 공통 `ConfirmModal`을 사용하여 삭제 전 확인
- 삭제 범위를 다음과 같이 명확하게 안내
  - 삭제 대상: 선택한 AI 여행 코스 문서
  - 유지 대상: 같은 폴더의 여행지 카드, 체크리스트, 일반 메모
- 삭제 성공 시 Firebase `aiTripPlans/{planId}`와 현재 화면의 코스 목록에서 즉시 제거
- 삭제 중 중복 요청과 상세 모달 종료를 방지

### 동일 조건 재생성

- 상세 화면에 `Regenerate` 기능 추가
- 저장된 코스의 생성 조건을 AI 플래너로 전달
- API를 즉시 호출하지 않고 플래너 입력 폼만 복원
- 사용자가 지역, 기간, 동행, 인원, 예산, 여행 속도, 날씨, 시간, 취향과 제외 조건을 다시 확인한 뒤 생성하도록 구성
- 위시리스트 폴더 기반 코스는 원본 폴더와 저장 장소 선택 상태도 함께 복원
- 이전에 저장된 코스처럼 생성 조건이 없는 문서는 현재 폴더를 기준으로 재생성 준비

자동 재호출 대신 확인 단계를 둔 이유는 불필요한 Gemini API 호출과 사용량 소비를 방지하고, 사용자가 조건을 조정할 기회를 제공하기 위해서입니다.

### 생성 조건 저장 구조 추가

새로 생성한 AI 코스를 저장할 때 `generation_context`를 함께 기록하도록 데이터 구조를 확장했습니다.

```text
generation_context
├─ planningMode
├─ sourceFolderId
├─ regionName
├─ durationDays
├─ companionType
├─ peopleCount
├─ budgetLevel
├─ pace
├─ weatherKeyword
├─ startTime
├─ endTime
├─ travelStyle
└─ avoidKeywords
```

기존 코스 데이터와의 호환성을 유지하기 위해 `generation_context`가 없는 문서도 정상 조회할 수 있으며, 새로 저장하는 문서부터 생성 조건이 누적됩니다.

## 22. AI 추천 장소의 관광공사 데이터 검증 보완

### 작업 배경

AI 여행 코스에 `contentId`가 포함되어 있으면 기존 저장 로직은 해당 값을 관광공사 데이터로 간주했습니다. 이 때문에 AI가 존재하지 않거나 다른 장소의 `contentId`를 반환하더라도 AI가 생성한 일반적인 장소명과 주소가 위시리스트 카드로 저장될 수 있었습니다.

예를 들어 `성북동 근처 한식 맛집`, `백화점 내 실내 카페`처럼 특정 관광공사 장소로 보기 어려운 이름에 `contentId`가 잘못 연결되면, 위시리스트 카드와 상세 페이지의 정보가 서로 달라질 가능성이 있었습니다.

### 저장 전 contentId 재검증

AI 코스를 폴더에 저장하기 직전에 각 장소의 `contentId`를 한국관광공사 `detailCommon2` API로 다시 조회하도록 변경했습니다.

- `contentId`가 실제 상세 조회 결과의 ID와 일치하는지 확인
- 상세 결과에 공식 장소명이 있는지 확인
- 검증 성공 시 AI 응답의 제목, 주소, 이미지 대신 관광공사 공식 정보 사용
- 동일한 `contentId`가 일정에 여러 번 등장하면 한 번만 조회하고 위시리스트 카드도 한 개만 저장
- 상세 API 응답은 기존 캐시 모듈을 사용하며 상세 데이터 캐시 기간은 14일 유지

### 검증 실패 장소 처리

검증되지 않은 장소를 삭제하지는 않고 AI 코스 문서에만 유지하도록 처리했습니다.

- 잘못된 `contentId` 또는 조회에 실패한 `contentId` 제거
- 원래 AI가 제안한 ID는 `aiSuggestedContentId`로 보관
- `source: ai_generated`, `tourApiVerified: false` 표시
- 위시리스트 여행지 카드로는 저장하지 않음
- 잘못된 관광공사 상세 페이지로 이동하는 링크가 만들어지지 않도록 방지

검증된 장소는 다음 정보를 저장합니다.

```text
source: tour_api
tourApiVerified: true
contentId: 관광공사 공식 contentId
placeName/title: 관광공사 공식 장소명
address/addr1: 관광공사 공식 주소
firstimage/imageUrl: 관광공사 공식 이미지
```

### 코스 문서 검증 통계

Firebase `aiTripPlans/{planId}`에 다음 검증 결과를 함께 기록하도록 확장했습니다.

```text
tour_api_verification
├─ verified_places
├─ document_only_places
├─ rejected_content_ids
└─ verified_at
```

- `verified_places`: 관광공사 검증 후 위시리스트 카드로 저장된 고유 장소 수
- `document_only_places`: 관광공사 카드로 저장하지 않고 코스 문서에만 유지한 일정 항목 수
- `rejected_content_ids`: AI가 제안했지만 관광공사에서 확인되지 않은 고유 ID 수

### 사용자 안내와 출처 표시

- 생성 직후 미저장 미리보기의 ID 보유 장소: `TourAPI 후보`
- 저장 후 검증된 장소: `TourAPI verified`
- 관광공사에서 확인되지 않은 장소: `AI 추천`
- 기존 저장 문서 중 검증 이력이 없는 ID 보유 장소: `TourAPI legacy`
- 저장 완료 알림에 관광공사 검증 여행지 수와 문서 전용 추천 장소 수를 구분하여 표시

### 기존 데이터 주의사항

이번 검증은 변경 이후 새로 저장하는 AI 코스부터 적용됩니다. 기존에 잘못된 제목이나 주소로 저장된 위시리스트 카드는 자동으로 변경하지 않았습니다. 기존 데이터는 별도 정리 또는 재저장이 필요하며, `TourAPI legacy` 표시를 통해 신규 검증 데이터와 구분할 수 있습니다.

## 23. AI 코스 저장 결과 불일치 및 상세 모달 UI 보정

### 작업 배경

로컬 테스트 중 동일한 AI 여행 코스를 두 번 저장했을 때 저장 결과가 서로 다르게 보이는 문제가 확인되었습니다.

- 대전 코스를 저장했는데 위시리스트 카드 영역에 `서울특별시 성북구` 장소가 함께 표시됨
- Firebase Realtime Database의 `aiTripPlans` 문서 구조 자체는 정상으로 보임
- 저장 성공 토스트 메시지가 한 줄로 너무 길게 표시됨
- AI 코스 문서 수정 모달에서 화면 폭이 줄어들면 본문 영역이 과도하게 좁아짐

### 원인 분석

AI 코스 저장 로직은 `contentId`가 실제 관광공사 상세 API에서 조회되는지까지 검증하도록 보완되어 있었지만, 다음 경계가 아직 부족했습니다.

1. 관광공사에 존재하는 `contentId`라도 현재 코스 지역과 다른 주소의 장소일 수 있음
2. 마이페이지 위시리스트 카드 목록은 폴더 ID 기준으로만 항목을 보여주기 때문에, 과거 저장된 잘못된 카드가 같은 폴더에 남아 있으면 계속 노출될 수 있음
3. `savedPlaces`는 고유 검증 장소 수이고, AI 코스 상세의 `PLACES`는 문서 내 전체 일정 항목 수라 사용자 입장에서 숫자 의미가 혼동될 수 있음

### 저장 전 지역 일치 검증 추가

AI 코스를 저장할 때 `generationContext.regionName`을 기준으로 관광공사 공식 주소를 한 번 더 비교하도록 수정했습니다.

- AI가 제안한 `contentId`로 `detailCommon2` 상세 조회
- 응답 `contentid`와 공식 `title` 존재 여부 확인
- 공식 주소 `addr1 + addr2`에서 광역 지역 키 추출
- 코스 생성 지역과 공식 주소 지역이 다르면 검증 실패 처리

예를 들어 대전 코스에서 서울 성북구 공식 주소가 조회되면 다음과 같이 처리합니다.

```text
관광공사 검증 위시리스트 카드 저장 안 함
AI 코스 문서 안의 AI 추천 장소로만 보관
```

### 마이페이지 카드 목록 방어 로직

화면 표시 단계에서도 한 번 더 방어하도록 보완했습니다.

- `source: ai_generated` 항목은 여행지 카드 목록에서 제외
- `aiSuggestedContentId`만 가진 미검증 항목은 여행지 카드 목록에서 제외
- AI 코스 폴더에 `verified_at`이 있는 카드가 있고, 코스 지역과 카드 주소 지역이 다르면 목록에서 제외

이번 조치는 신규 저장 데이터뿐 아니라 이미 폴더에 남아 있는 일부 과거 데이터가 화면에서 섞여 보이는 문제도 완화하기 위한 보정입니다. 단, 기존 Firebase 데이터 자체를 자동 삭제하지는 않았습니다.

### 저장 완료 토스트 메시지 보정

저장 결과 메시지가 너무 길게 한 줄로 표시되어 화면을 가로로 크게 차지하던 문제를 수정했습니다.

## 24. 위시리스트 삭제 로직 및 통계 라벨 보완

### 작업 배경

AI 코스 저장 검증 이후 마이페이지 위시리스트 카드에서 과거 테스트 카드가 화면에 남아 있을 수 있는지 확인했습니다. 새로 저장한 대전 코스에서는 서울 카드가 더 이상 생성되지 않았고, Realtime Database에도 잘못된 서울 카드 데이터가 남아 있지 않은 것을 확인했습니다.

다만 위시리스트 카드 삭제 버튼을 눌러도 일부 항목이 삭제되지 않는 문제가 있었고, 삭제 확인 팝업이 브라우저 기본 confirm으로 표시되어 현재 프로젝트 UI 톤과 맞지 않았습니다.

### 원인

기존 삭제 흐름은 위시리스트 항목의 고유 키를 `contentId` 하나로만 비교하는 구조였습니다. Firebase 전환과 AI 코스 저장 구조 개선을 거치면서 일부 데이터에는 `contentid`, `contentId`, `content_id`가 섞여 저장될 수 있었기 때문에, 화면에서는 카드가 보이지만 삭제 로직에서는 같은 항목을 찾지 못하는 경우가 생겼습니다.

또한 기존 `toggleWishlist`를 삭제에도 재사용하고 있어, 삭제 대상 식별이 실패하면 사용자 입장에서는 삭제 버튼을 눌렀는데도 항목이 남아 있는 것처럼 보일 수 있었습니다.

### 수정

- 위시리스트 항목의 관광공사 ID를 `getTourContentId` 기준으로 정규화
- `contentId`, `contentid`, `content_id` 형태를 모두 같은 값으로 비교
- 삭제 전용 API `removeWishlistItem` 추가
- Zustand 위시리스트 스토어에 `removeWishlistItem` 액션 추가
- 마이페이지 카드 삭제 버튼이 삭제 전용 액션을 호출하도록 변경
- 삭제 확인을 브라우저 기본 팝업에서 프로젝트 공통 `ConfirmModal`로 교체
- 삭제 성공 시 초록색 성공 토스트 표시

### 위시리스트 통계 라벨 보완

마이페이지 위시리스트 사이드바의 라벨도 사용자에게 더 직관적으로 보이도록 정리했습니다.

- `TOTAL_NODES` → `SAVED_PLACES`
- `ALL_NODES` → `ALL_PLACES`
- `TOP_FOLDER` → `MOST_SAVED_FOLDER`
- `FOLDERS` 제목 옆에 실제 폴더 개수 배지 추가

`ALL_PLACES`는 폴더 개수가 아니라 전체 저장 여행지 카드 수를 의미합니다. 폴더 개수는 `FOLDERS` 배지와 상단 통계의 `FOLDERS` 값으로 별도 표시합니다.

### 위시리스트 헤더 톤 정리
