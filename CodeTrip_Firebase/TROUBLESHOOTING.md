## 9. AI가 반환한 TourAPI contentId와 장소 정보가 일치하지 않는 문제

- **발생일**: 2026-07-24
- **영향 범위**: AI 여행 코스 저장, 위시리스트 여행지 카드, 여행지 상세 페이지 연결
- **후속 보정**: 관광공사 상세 조회가 성공하더라도 코스 지역과 공식 주소 지역이 다르면 위시리스트 카드 저장 대상에서 제외

### 증상

AI 여행 코스 하단에 다음과 같이 특정 관광지로 보기 어려운 일반적인 장소명이 위시리스트 카드로 저장되었습니다.

- `성북동 근처 한식 맛집`
- `백화점 내 실내 카페`

카드에는 이미지와 `VIEW DATA` 버튼이 표시되어 관광공사 API에 등록된 장소처럼 보였지만, 실제로는 AI가 응답에 포함한 `contentId`를 애플리케이션이 별도 검증 없이 신뢰한 결과였습니다. 잘못된 ID가 연결되면 카드 제목과 상세 페이지의 관광공사 장소 정보가 서로 다를 수 있었습니다.

이후 로컬 테스트에서 동일한 코스를 여러 번 저장했을 때 대전 코스 폴더에 `서울특별시 성북구` 장소 카드가 함께 보이는 사례도 확인했습니다. Firebase `aiTripPlans` 문서 구조는 정상으로 보였기 때문에, 문제의 핵심은 코스 문서 저장 자체보다 **검증된 위시리스트 카드로 승격할 수 있는 장소의 판정 기준**에 있었습니다.

### 근본 원인

기존 `saveAiTripToFolder` 저장 로직은 다음 조건만 확인했습니다.

```text
AI 일정 항목에 contentId가 존재하는가?
└─ 예: 위시리스트 여행지 카드로 저장
```

하지만 생성형 AI의 구조화 응답은 형식이 올바르더라도 값의 사실성까지 보장하지 않습니다. 즉 다음 상황을 구분하지 못했습니다.

1. 관광공사에 실제로 존재하는 올바른 `contentId`
2. 다른 장소의 `contentId`가 잘못 연결된 경우
3. 관광공사에 존재하지 않는 ID를 AI가 생성한 경우
4. 공식 장소가 아닌 식사·카페 설명에 임의 ID가 연결된 경우

AI 응답의 장소명, 주소와 이미지도 그대로 저장했기 때문에 `contentId`만 관광공사 형식이고 나머지 내용은 AI가 생성한 혼합 데이터가 만들어질 수 있었습니다.

### 해결 방향

AI 응답의 `contentId`는 확정 데이터가 아니라 **검증 후보**로 취급하도록 저장 경계를 변경했습니다.

```text
AI 코스 생성
  ↓
contentId 후보 수집 및 중복 제거
  ↓
관광공사 detailCommon2 재조회
  ↓
ID 일치 + 공식 장소명 존재 여부 확인
  ├─ 검증 성공: 공식 정보로 위시리스트 카드 저장
  └─ 검증 실패: AI 코스 문서에만 유지
```

### 검증 성공 판정

다음 조건을 모두 만족해야 관광공사 검증 장소로 처리합니다.

- `detailCommon2` 상세 조회가 성공함
- 응답의 `contentid`가 AI가 전달한 `contentId`와 정확히 일치함
- 응답에 비어 있지 않은 공식 `title`이 존재함
- 공식 주소의 광역 지역이 코스 생성 지역과 일치함

검증에 성공하면 AI가 반환한 장소 정보를 사용하지 않고 다음 공식 데이터로 교체합니다.

- 장소명: `detailCommon2.title`
- 주소: `addr1 + addr2`
- 대표 이미지: `firstimage` 또는 `firstimage2`
- 콘텐츠 유형: `contenttypeid`

위시리스트 카드에는 `source: tour_api`, `verified_at`을 기록하며, 코스 문서의 일정 항목에는 `tourApiVerified: true`를 기록합니다.

### 지역 불일치 후속 보정

`contentId`가 관광공사 상세 API에서 조회되더라도, AI가 현재 코스와 무관한 다른 지역의 ID를 연결할 수 있습니다. 예를 들어 대전 코스에 서울 성북구 장소 ID가 섞이면 API 검증 자체는 성공하지만 사용자 입장에서는 잘못된 저장 결과가 됩니다.

이를 막기 위해 저장 시점에 다음 비교를 추가했습니다.

```text
generationContext.regionName
  ↓
코스 기준 광역 지역 추출
  ↓
detailCommon2 공식 주소 addr1 + addr2
  ↓
공식 주소 광역 지역 추출
  ↓
지역이 다르면 TourAPI verified 저장 대상에서 제외
```

현재는 서울, 부산, 대구, 인천, 광주, 대전, 울산, 세종, 경기, 강원, 충북, 충남, 전북, 전남, 경북, 경남, 제주 기준으로 광역 지역을 비교합니다. 더 세밀한 시/군/구 단위 동선 검증은 후속 개발 계획으로 분리합니다.

지역이 일치하지 않는 장소는 다음처럼 처리합니다.

- 위시리스트 여행지 카드로 저장하지 않음
- AI 코스 문서에는 일정 항목으로 유지
- `source: ai_generated`, `tourApiVerified: false`로 낮춤
- AI가 제안한 ID는 `aiSuggestedContentId`로만 보관

### 화면 표시 방어

이미 과거에 저장된 카드가 같은 폴더에 남아 있으면 새 저장 로직을 고쳐도 화면에서 계속 섞여 보일 수 있습니다. 기존 데이터 삭제는 사용자가 직접 저장한 장소와 자동 저장된 장소를 완전히 구분하기 어려워 자동 처리하지 않았습니다.

대신 마이페이지 카드 목록에서 다음 항목은 표시하지 않도록 보정했습니다.

- `source: ai_generated`
- `aiSuggestedContentId`만 보유한 미검증 장소
- `verified_at`이 있으나 AI 코스 지역과 카드 주소 지역이 다른 장소

이 조치는 기존 잘못된 데이터가 화면에 계속 노출되는 문제를 줄이기 위한 방어 로직이며, Firebase 데이터 자체를 삭제하지는 않습니다.

### UI 관련 후속 보정

동일 테스트 과정에서 함께 발견된 UI 문제도 보완했습니다.

- AI 코스 저장 완료 토스트가 너무 긴 한 줄로 표시됨
  - 최대 너비 적용
  - 줄바꿈 표시 지원
  - 저장 성공 문구를 2줄 구조로 변경
- AI 코스 문서 수정 모달이 작은 화면에서 과도하게 좁아짐
  - 모바일에서는 모달 전체가 세로 스크롤되도록 변경
  - 요약 입력 textarea 최소 높이 확보
  - 데스크톱에서는 기존 본문/메타 2단 레이아웃 유지

### 검증 실패 및 API 장애 처리

다음 경우에는 위시리스트 카드를 생성하지 않습니다.

- 상세 조회 결과가 없음
- 요청한 ID와 응답 ID가 다름
- 공식 장소명이 없음
- 관광공사 API 요청이 실패함

여행 코스 자체가 사라지지 않도록 해당 일정은 AI 코스 문서에 유지합니다. 다만 잘못된 상세 페이지 연결을 방지하기 위해 `contentId`, `contentid`, `content_id`는 제거하고 다음 메타데이터를 기록합니다.

```text
source: ai_generated
tourApiVerified: false
aiSuggestedContentId: AI가 원래 제안한 ID
```

관광공사 API 장애가 발생해도 전체 폴더 저장을 실패시키지 않고, 검증하지 못한 장소를 문서 전용 데이터로 낮추는 방식으로 처리했습니다. 이 방식은 잘못된 카드를 저장하는 것보다 보수적인 동작입니다.

### 중복 호출과 저장 수 보정

동일한 장소가 여러 일정에 반복될 수 있으므로 `contentId` 기준으로 후보를 먼저 중복 제거했습니다.

- 관광공사 상세 API는 고유 ID별로 한 번만 호출
- 기존 상세 조회 캐시의 14일 TTL 활용
- 위시리스트 카드도 고유 장소별 한 개만 저장
- 일정 문서에는 각 시간대의 원래 배치를 모두 유지

기존에는 일정 항목 수를 저장된 여행지 수처럼 표시할 수 있었지만, 변경 후에는 실제로 저장된 고유 검증 카드 수를 `savedPlaces`로 반환합니다.

### Firebase 저장 구조

새 AI 코스 문서에는 검증 결과 요약을 함께 기록합니다.

```text
users/{uid}/aiTripPlans/{planId}/tour_api_verification
├─ verified_places
├─ document_only_places
├─ rejected_content_ids
└─ verified_at
```

- `verified_places`: 검증 후 위시리스트 카드로 저장된 고유 장소 수
- `document_only_places`: 코스 문서에만 유지된 일정 항목 수
- `rejected_content_ids`: 관광공사에서 확인되지 않은 고유 ID 수

Realtime Database Rules의 위시리스트 검증 조건은 필수 필드 존재 여부와 사용자 UID를 검사하며 추가 메타데이터를 금지하지 않으므로, `source`, `verified_at` 추가로 인한 규칙 변경은 필요하지 않았습니다.

### 사용자 안내 개선

데이터의 검증 단계를 화면에서 구분하도록 배지를 세분화했습니다.

- `TourAPI 후보`: AI 생성 직후이며 아직 저장 전 검증을 거치지 않은 장소
- `TourAPI verified`: 저장 과정에서 관광공사 공식 정보가 확인된 장소
- `AI 추천`: 관광공사에서 확인되지 않아 코스 문서에만 남은 장소
- `TourAPI legacy`: 기존 문서 중 `contentId`는 있지만 신규 검증 이력이 없는 장소

저장 완료 알림도 관광공사 검증 여행지 수와 코스 문서 전용 장소 수를 분리해 알려주도록 변경했습니다.

### 기존 데이터 처리 범위

이번 변경은 새로 저장하는 AI 코스부터 적용됩니다. 이미 잘못된 제목이나 주소로 저장된 위시리스트 카드는 자동 수정하지 않았습니다.

- 기존 코스 문서: `TourAPI legacy`로 표시
- 기존 위시리스트 카드: 필요 시 수동 삭제 후 코스 재생성·재저장
- 자동 마이그레이션 미적용 이유: 기존 카드의 생성 출처와 사용자가 직접 저장한 장소를 안전하게 구분하기 어려움

### 검증 결과


## 10. 위시리스트 카드 삭제가 일부 항목에서 동작하지 않은 문제

- **발생일**: 2026-07-24
- **영향 범위**: 마이페이지 위시리스트 카드 삭제, 위시리스트 통계 표시
- **요약**: 위시리스트 데이터의 관광공사 ID 필드명이 `contentId`, `contentid`, `content_id`로 혼재되어 일부 카드 삭제 대상이 제대로 매칭되지 않던 문제를 삭제 전용 로직과 ID 정규화로 보완

### 증상

마이페이지 위시리스트 카드에서 하트 삭제 버튼을 눌러도 일부 카드가 삭제되지 않았습니다. 특히 AI 코스 저장 구조 보완 이전에 생성된 카드나 서로 다른 저장 경로를 거친 카드에서 문제가 발생할 수 있었습니다.

또한 삭제 확인 메시지가 브라우저 기본 confirm으로 표시되어 현재 프로젝트의 모달/토스트 UI 톤과 맞지 않았습니다.

### 원인

위시리스트 항목은 저장 시점과 저장 경로에 따라 관광공사 ID 필드명이 다음처럼 달라질 수 있었습니다.

```text
contentId
contentid
content_id
```

기존 삭제 로직은 특정 필드명만 기준으로 항목을 찾았습니다. 따라서 화면에는 카드가 표시되지만 삭제 API에서는 같은 항목으로 인식하지 못하는 경우가 생겼습니다.

### 해결

- `getTourContentId` 기준으로 위시리스트 항목의 ID를 정규화
- `toggleWishlist` 삭제 재사용 대신 `removeWishlistItem` 삭제 전용 API 추가
- 같은 `contentId` 계열 값을 가진 위시리스트 항목을 찾아 Firebase update로 제거
- 삭제 후 Zustand 스토어를 다시 동기화해 화면 카운트와 카드 목록 갱신
- 삭제 확인 UI를 브라우저 기본 confirm에서 `ConfirmModal`로 교체

### 라벨 혼동 보완

테스트 과정에서 `ALL_NODES`가 폴더 개수처럼 보일 수 있다는 혼동이 확인되어 라벨도 함께 정리했습니다.

## 11. CodeRabbit 리뷰 대응 중 빌드 권한 및 위시리스트 성공 토스트 오판 문제

- **발생일**: 2026-07-24
- **영향 범위**: 위시리스트 추가/삭제 토스트, AI 코스 문서 렌더링, Gemini API 호출 안정성, 로컬 빌드 검증
- **요약**: CodeRabbit 리뷰를 통해 위시리스트 토글 실패 시에도 성공 메시지가 표시될 수 있는 구조와 `plan.days` 비정상 데이터 렌더링 위험, Gemini API 네트워크 예외 처리 부족 문제가 확인되었습니다. 수정 후 빌드 검증 과정에서는 샌드박스 쓰기 권한 제한으로 Vite 임시 파일 생성이 차단되는 문제가 있었습니다.

### 증상

위시리스트 토글 함수가 삭제 성공과 오류를 모두 `false`로 반환하고 있어, 호출부에서 실제 실패와 정상 삭제를 구분하기 어려운 상태였습니다. 이 구조에서는 Firebase 요청이 실패했는데도 사용자에게 성공 토스트가 표시될 가능성이 있었습니다.

AI 코스 문서 화면에서는 `plan.days`가 배열이라는 전제를 두고 렌더링하는 지점이 있어, 오래된 데이터나 비정상 데이터가 들어오면 `TypeError`로 화면 렌더링이 중단될 수 있었습니다.

Gemini API 호출은 HTTP status 기반 재시도만 처리하고 있었고, 네트워크 오류, DNS 실패, 오프라인, 요청 지연 timeout에 대한 catch 및 재시도 처리가 부족했습니다.

### 원인

- `toggleWishlist`가 boolean만 반환해 성공/실패 상태를 충분히 표현하지 못함
- 일부 호출부가 반환값을 확인하지 않고 바로 성공 토스트를 표시함
- AI 코스 문서 데이터가 항상 최신 정상 구조라는 가정이 남아 있음
- `fetch()` 자체가 reject되는 네트워크 예외와 timeout 제어가 없음
- Codex 실행 환경의 읽기 전용 샌드박스에서는 Vite가 `node_modules/.vite-temp`에 임시 config 파일을 생성할 수 없음

### 해결

- `toggleWishlist` 반환값을 `{ success, wishlisted, error }` 구조로 변경
- 위시리스트 추가/삭제 호출부에서 `success`와 `wishlisted`를 확인한 뒤 성공 토스트 표시
- AI 코스 문서 렌더링에 `getPlanDays(plan)` 헬퍼 추가
- `fetchGeminiWithRetry`에 `AbortController` 기반 45초 timeout 추가
- timeout, 네트워크 오류, 재시도 가능 HTTP status를 구분해 재시도 처리
- 일반 권한 빌드에서 `EPERM` 발생 확인 후, 쓰기 권한이 있는 실행으로 `npm run build` 재검증

## 12. 같은 AI 코스를 여러 폴더에 저장하면 기존 폴더의 장소가 사라지는 문제

- **발생일**: 2026-07-24
- **영향 범위**: AI 여행 코스 저장, 위시리스트 폴더별 장소 카운트, 마이페이지 폴더 목록
- **요약**: 같은 조건으로 AI 여행 코스를 두 번 생성하고 각각 다른 폴더에 저장했을 때, 첫 번째 폴더에 저장되어 있던 관광공사 검증 여행지 카드가 두 번째 폴더로 이동되어 첫 번째 폴더의 장소 수가 `0`으로 표시되는 문제가 있었습니다.

### 증상

사용자가 같은 조건으로 AI 여행 코스를 여러 번 생성한 뒤 각각 다른 폴더에 저장하면, 이전 폴더에 있던 여행지 카드가 사라진 것처럼 보였습니다.

실제 Realtime Database에서는 여행지 카드가 삭제된 것이 아니라, 같은 `contentId`를 가진 기존 위시리스트 카드의 `folder_id`가 새 폴더 ID로 갱신되어 있었습니다. 이 때문에 기존 폴더 카운트가 감소하고 새 폴더에 카드가 몰리는 현상이 발생했습니다.

### 원인

AI 코스 저장 함수에서 관광공사 검증 장소를 위시리스트 카드로 저장할 때 기존 항목을 찾는 기준이 `contentId` 하나뿐이었습니다.

```text
기존 기준:
contentId가 같으면 같은 위시리스트 카드로 판단
```

하지만 CodeTrip의 AI 코스 폴더 구조에서는 같은 장소가 여러 여행 코스 폴더에 동시에 포함될 수 있습니다. 즉, `contentId`가 같더라도 `folder_id`가 다르면 서로 다른 카드로 저장되어야 합니다.

### 해결

AI 코스 저장 로직의 기존 카드 조회 기준을 `contentId + folder_id` 조합으로 변경했습니다.

```text
변경 기준:
contentId가 같고 folder_id도 같은 경우에만 같은 위시리스트 카드로 판단
```

이제 같은 관광공사 장소라도 다른 AI 코스 폴더에 저장될 경우 새 위시리스트 카드로 생성됩니다. 동일 폴더에 같은 장소가 다시 저장될 때만 기존 카드를 재사용합니다.

기존 폴더에서 장소가 빠져 있다면 다음 중 하나로 정리합니다.

3. Realtime Database에서 명확히 잘못 이동된 카드만 확인 후 수동 정리

## 13. 위시리스트 폴더 선택 시 다른 폴더 여행지가 함께 표시되는 문제

- **발생일**: 2026-07-24
- **영향 범위**: 마이페이지, 위시리스트 폴더별 카드 목록, AI 코스 저장 결과 확인
- **요약**: 특정 위시리스트 폴더를 선택했을 때 해당 폴더에 저장된 여행지만 보여야 하지만, 다른 폴더에 연결된 여행지 카드가 함께 노출되는 문제가 있었습니다.

### 증상

예를 들어 `서울 혼자 맛집 탐방` 폴더를 선택했는데, 카드 하단의 `FOLDER:` 라벨에는 `서울 비오는날 실내 문화 코스`, `서울 실내 문화 친구 여행`처럼 다른 폴더명이 표시되는 여행지 카드가 함께 보였습니다.

### 원인

마이페이지의 위시리스트 목록은 기본적으로 `folder_id`로 필터링하고 있었지만, 이후 최근 AI 코스 문서의 지역 정보를 기준으로 여행지를 다시 필터링하는 전역 로직이 추가되어 있었습니다.

이 로직은 AI 코스 문서의 지역과 관광공사 검증 장소를 맞추기 위한 의도로 보이나, 폴더 선택 화면에서는 다음 문제가 있었습니다.

- 폴더 목록은 폴더 단위 저장 상태를 보여줘야 함
- AI 코스 문서의 지역 정보는 현재 선택한 폴더의 카드 표시 기준이 아님
- 여러 AI 코스 문서가 쌓이면 최근 코스 지역이 다른 폴더 화면에도 영향을 줄 수 있음

### 해결

마이페이지의 여행지 카드 목록은 선택한 폴더의 `folder_id`만 기준으로 필터링하도록 정리했습니다.

```text
선택 폴더 있음: item.folder_id === selectedFolderId
미분류 선택: item.folder_id 없음
전체 선택: 전체 저장 여행지
```

추가로 관광공사 검증을 통과하지 못한 AI 추천 장소는 기존처럼 카드 목록에서 제외했습니다.

```text
source === 'ai_generated' 또는 aiSuggestedContentId가 있는 항목은 카드 목록 제외
```

이제 AI 코스 문서 안의 추천 장소와 실제 위시리스트 카드 목록의 역할이 분리됩니다.

### 수동 확인 항목

1. 특정 폴더 선택 시 표시되는 모든 카드의 `FOLDER:` 라벨이 선택 폴더와 일치하는지 확인
2. `ALL_PLACES` 선택 시 전체 저장 여행지가 정상 표시되는지 확인
3. AI 코스 문서는 폴더 상단 문서 영역에 표시되고, 검증되지 않은 AI 추천 장소가 카드로 섞이지 않는지 확인
4. 기존에 잘못 저장된 카드가 Realtime Database에 남아 있다면 해당 카드의 `folder_id`를 확인 후 수동 삭제 또는 이동

### 추가 원인: 같은 contentId를 가진 카드의 React key 충돌

폴더 필터링 기준을 정리한 뒤에도 화면에서 다른 폴더 카드가 섞여 보일 수 있는 추가 원인을 확인했습니다.

AI 코스 저장 이후에는 같은 관광공사 장소가 여러 폴더에 각각 저장될 수 있습니다. 이때 저장 항목은 서로 다른 Firebase wishlist item이지만, 관광공사 `contentId`는 같습니다.

기존 마이페이지 카드 렌더링은 React `key`로 `contentId`만 사용했습니다.

```text
기존 key: contentId
문제: 같은 장소가 여러 폴더에 있으면 서로 다른 카드가 같은 key를 공유
```

React는 같은 key를 가진 요소를 동일한 항목으로 재사용할 수 있기 때문에, 폴더 전환 시 이전 폴더의 카드 DOM이 섞여 보이는 현상이 발생할 수 있었습니다.

수정 후에는 Firebase wishlist item의 고유 ID를 우선 key로 사용합니다.

```text
변경 key: wishlist item id
fallback: contentId + folder_id
```

이제 같은 관광공사 장소가 여러 폴더에 저장되어도 각 카드가 서로 다른 UI 항목으로 렌더링됩니다.

## 14. AI 추천 장소가 위시리스트 카드 목록에 보이지 않는 현상

- **발생일**: 2026-07-24
- **영향 범위**: 마이페이지 위시리스트 폴더, AI 코스 문서, 여행지 카드 목록
- **요약**: AI 코스 문서에는 존재하는 장소가 하단 위시리스트 카드 목록에는 표시되지 않아 데이터가 삭제된 것처럼 보일 수 있었습니다.

### 증상

AI 여행 코스 문서에는 `AI 추천` 배지가 붙은 장소가 남아 있지만, 같은 폴더의 하단 여행지 카드 목록에는 해당 장소가 보이지 않았습니다.

### 원인

의도된 데이터 분리 정책입니다.

AI가 생성한 장소 중 한국관광공사 API 검증에 성공한 항목은 `TourAPI verified`로 저장되어 위시리스트 카드에 표시됩니다. 반면 관광공사 데이터와 정확히 매칭되지 않은 장소는 `AI 추천` 항목으로 분류되어 위시리스트 카드가 아닌 AI 코스 문서 내부에만 보관됩니다.

### 판단 기준

```text
TourAPI verified
- 관광공사 contentId 있음
- 위시리스트 카드로 표시
- VIEW_DATA 상세 이동 가능

AI 추천
- 관광공사 검증 실패 또는 contentId 없음
- 위시리스트 카드로 표시하지 않음
- AI 코스 문서 내부 일정 항목으로만 표시
```

### 결론

데이터가 삭제된 것이 아닙니다. 위시리스트 카드 목록과 AI 코스 문서의 표시 범위를 분리한 결과입니다.

이 기준을 유지하면 사용자는 카드 목록에서 실제 관광공사 기반 여행지만 확인할 수 있고, AI가 보완 생성한 장소는 코스 문서 맥락 안에서 확인할 수 있습니다.
